# Star-link2

这是一个从零开始搭建的竖屏网页对战小游戏原型，当前重点不是完整玩法，而是先把界面语言和代码结构定稳，方便后续持续往里加战斗内容。

## 当前设计理念

### 1. 先做竖屏战斗感

- 整个页面优先按手机竖屏体验设计。
- 对手信息固定在左上，我方信息固定在左下，中间区域留给后续战斗表现。
- 背景铺满横向宽度，但主体内容仍保持窄屏战斗视图，避免桌面端看起来太散。
- 当前视觉风格统一使用像素风，后续新增 UI、面板、按钮、卡牌和特效默认都要延续像素风语言，不再回到圆润毛玻璃或现代卡片风。

### 2. 角色信息尽量紧凑

- 头像使用像素徽章式方形底座，强调像素风识别感。
- 血条和能量条贴近头像，减少 UI 占用面积。
- 生命值使用红色，能量使用雅黄色，直接传达资源类型。
- 数字直接显示在条内，保证玩家不需要额外读文字说明。
- `HP` 和 `EP` 标签放在右侧，信息顺序更接近“先看状态条，再看类型”。

### 3. 前端只做展示，核心计算放服务端

- 前端负责渲染界面、显示数值、发送操作请求。
- 核心战斗结算不放在浏览器内联脚本里，避免逻辑全暴露在页面源码中。
- 用户即使能看到前端脚本，也只能看到展示层和接口调用，关键规则仍在服务端。

### 4. 战斗规则按类型拆分，避免越写越乱

- 状态初始化单独放一处。
- 战斗引擎负责调度动作，不直接堆具体伤害细节。
- 血量和能量结算拆成独立 effect 模块。
- 每一种动作单独一个文件，后续新增技能、法术、卡牌效果时直接扩展，不把所有逻辑塞进一个大文件。

这套拆分的目的很明确：后面继续做时，新增内容应该是“加文件”，而不是“把旧文件越改越臃肿”。

## 当前文件结构

```text
.
├── index.html                  # 页面结构与样式
├── app.js                      # 前端渲染与接口调用
├── server.js                   # HTTP 服务与 API 路由
└── src/battle
    ├── constants.js            # 战斗常量
    ├── state.js                # 初始状态定义
    ├── engine.js               # 战斗调度入口
    ├── actions
    │   └── basic-strike.js     # 当前示例动作
    └── effects
        ├── health.js           # 生命相关结算
        └── energy.js           # 能量相关结算
```

## 后续扩展原则

- 新动作放进 `src/battle/actions/`
- 新资源或结算规则放进 `src/battle/effects/`
- 页面布局尽量保持紧凑，不轻易回到大块面板式 UI
- 前端视觉统一保持像素风：硬边框、像素阴影、低频色块和复古配色优先
- 能放服务端的规则尽量放服务端，前端只保留表现层

## 当前状态

目前已经具备：

- 双方初始生命值 `100`
- 双方初始能量 `10`
- 页面可通过接口同步战斗状态
- 抽卡、出牌、重置等基础战斗操作
- 我方手牌支持拖拽出牌
- 示例动作 `basic-strike` / `play-card`
- 抽卡入手动画，且已完成首张落点、多手牌让位与最终落点校正
- 出牌碎裂与粒子吸入回收区特效
- 能量不足时会明确提示，并让卡牌动画回到原手牌位置
- 重置战斗状态

接下来可以继续往这个结构里加技能、回合制规则、动画表现、卡牌系统或 AI 行为。
