# Current Progress

## 当前项目状态

项目当前已经从“基础卡牌原型”迭代成一版可玩的竖屏像素风卡牌战斗 Demo，重点不再是把元素摆出来，而是统一视觉语言、压交互手感，并把状态反馈做成可扩展的表现层系统。

当前本地访问：

- `http://127.0.0.1:9910/`

## 当前 UI 状态

### 1. 整体视觉方向

- 主视觉已统一为老式像素卡牌风
- 卡牌圆角已明显收小，整体更硬朗
- 卡牌正面新增像素外框
- 卡背不加外框，和正面区分开
- 右侧技能说明面板也已转成像素风语义
- 页面整体仍保持竖屏窄战场布局

### 2. 状态栏与战场布局

- 敌方状态栏固定在顶部
- 我方状态栏固定在底部左侧
- 我方状态栏宽度已收窄，为右侧技能说明区腾空间
- 生命条与能量条高度已加高
- 条内数值显示已修正，不再被裁切
- 敌方意图显示已移除

### 3. 手牌与详情区

- 我方手牌已上移，避免遮挡右侧详情区
- 详情区显示“最后一次选中的卡”
- 当前手牌卡面只展示：
  - 名字
  - 名字旁的能量消耗
  - 卡图
- 卡面上的独立效果 UI 已删除
- 伤害值不再单独显示在卡面上
- 效果说明统一放在右侧技能说明区

## 当前战斗交互状态

### 1. 手牌操作

- 我方手牌支持拖拽出牌
- 卡牌拖过战场中线后松手，判定为出牌
- 未过中线会回弹
- 已修复“松手后卡牌不弹回”的问题
- 当前高亮和最后选中状态可共存，不会丢失详情展示

### 2. 战斗基础流程

- 前端通过接口同步服务端战斗状态
- 支持重置
- 支持抽卡
- 支持我方出牌
- 支持敌方自动行动与对应演出
- 状态变化后，前端根据前后状态差异驱动表现

### 3. 当前状态事件来源

当前前端已根据状态差异识别以下事件：

- `damage`
- `shield`
- `energy`

这些事件不是绑在某张卡上，而是从状态变化里推导出来的。  
这意味着后续即使出现“敌方获得护盾”“敌方回能”“我方受治疗”这类效果，只要状态变了，表现层就可以继续沿用同一套机制。

## 当前视觉反馈系统

### 1. 顶层反馈层

- 飘字已放到最上层 UI，不会被其它节点遮挡
- 粒子画布不是整屏 `canvas`
- 粒子特效只创建局部范围画布，避免低端设备掉帧
- 当前粒子数量已固定压到 `100`

### 2. 状态反馈时序

当前状态反馈已经按“命中时机”触发，而不是状态一改就立即闪：

- 伤害反馈延迟到攻击粒子命中后触发
- 护盾反馈延迟到护盾粒子命中后触发
- 回能反馈延迟到回能粒子命中后触发

当前状态反馈包含：

- 角色受击震动
- 护盾高亮
- 能量高亮
- 飘字提示
- 命中爆点

### 3. 粒子飞行目标

粒子不再固定飞向回收区，而是按语义飞向目标区域：

- 伤害 -> 目标生命区域
- 护盾 -> 目标护盾区域
- 回能 -> 目标能量区域
- 如果后续某类事件没有明确目标节点，仍可兜底飞向回收区

这套目标分发同时支持：

- 我方状态变化
- 敌方状态变化
- 未来的“给敌人加盾”“给敌人回能”等情况

## 当前粒子与命中系统

### 1. 已完成的粒子样式

当前粒子已按效果类型拆成独立视觉：

- `damage` -> 像素斩击粒子
- `shield` -> 像素护盾粒子
- `energy` -> 像素火花粒子
- `default` -> 通用像素块粒子

### 2. 已完成的命中爆点样式

当前命中爆点也已按效果类型拆分：

- `damage` -> 斩击爆点
- `shield` -> 护盾包裹式爆点
- `energy` -> 能量火花爆点
- `default` -> 通用像素爆点

### 3. 当前扩展结构

当前表现系统已经拆成两层注册式结构：

1. `getParticleEffectProfile(effectType)`
   - 决定轨迹参数、时序参数、反馈延迟、粒子样式、命中爆点样式
2. 渲染注册表
   - `PARTICLE_RENDERERS`
   - `IMPACT_RENDERERS`

当前扩展新效果的标准做法：

1. 新增一种状态事件类型或表现语义
2. 在 `getParticleEffectProfile()` 里补对应 profile
3. 在渲染注册表里补粒子样式和命中爆点样式
4. 让状态变化描述符能找到目标节点

主动画循环不需要重写。

## 当前代码侧关键能力

### 1. 状态差异驱动表现

前端已具备以下能力：

- 从前后状态推导 `damage / shield / energy` 事件
- 为事件找到对应 DOM 目标区域
- 为事件生成带权重的目标描述符
- 将卡牌碎裂粒子分配到对应状态目标
- 将状态反馈延迟对齐到粒子命中阶段

### 2. 性能优化现状

- 粒子数量已压到 `100`
- 不再使用整屏 `canvas`
- 画布范围按卡牌和目标区域局部计算
- 粒子为像素块风格，避免高成本发光球
- 运动采用像素步进感，而不是高频平滑漂浮
- 当前目标是在低端设备上优先保证稳定，再继续加爽感

## 当前规则层状态

- 前后端仍是“前端负责展示，服务端负责战斗结算”
- 当前已支持出牌、抽卡、回合推进的基础链路
- 现有前端表现已经不再依赖“某张牌写死某种特效”，而是优先依赖状态变化结果

## 当前仍可继续优化的方向

### 高优先级

1. 继续细分命中爆点风格，例如斩击、穿刺、爆裂
2. 为不同技能类型增加独立状态事件，例如治疗、吸能、中毒
3. 为非状态类效果增加独立目标语义，例如回收区、牌库、手牌区
4. 把更多视觉参数收进统一配置，减少散落常量

### 中优先级

1. 继续压低低端设备上的掉帧风险
2. 提升不同效果的区分度，避免“看见很多粒子但不知道发生了什么”
3. 让命中爆点与状态栏闪烁的节奏更统一

## 当前结论

这一版已经不只是“卡牌能打出去”，而是形成了统一方向：

- UI 已经从通用卡牌样式转成偏老式像素卡
- 卡牌信息展示被压缩到更合理的层级
- 状态反馈已开始从“卡片特效”过渡到“状态变化表现系统”
- 粒子、命中爆点、受击反馈都具备继续扩展的基础接口

后续继续做时，优先应该新增“效果语义”和“表现 profile”，而不是回到把特效写死在单张卡上的方式。
